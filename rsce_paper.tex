\begin{abstract}
We present \textit{Residual Stream Context Encoding} (RSCE), a training-free
method for eliminating repeated long-context token costs in retrieval-augmented
inference. Given a context document $\ctx$, RSCE extracts a single vector
$C \in \mathbb{R}^{d_M}$ by mean-pooling the residual stream activations at a
calibrated layer $f(M)$, then injects $C$ as an additive shift into the residual
stream at query time, substituting the full $|T(\ctx)|$-token prefill with an
$O(1)$ operation. A compact explicit fact block $F$ handles factual precision
cases where latent compression is lossy. The encoded vector is computed once per
context and reused across all subsequent queries, amortizing setup cost over
$N \geq 2$ queries.

We evaluate RSCE on two domains across four decoder-only architectures spanning
7B--24B parameters: (1) multi-document QA (HotpotQA + 2WikiMultiHopQA from
LongBench), and (2) cross-file code completion (RepoBench-C). On QA with
LLaMA-3 8B ($n = 42$), Vec+F achieves Token F1 of $0.218$ versus baseline
$0.165$ at $\mathbf{97.8\%}$ input token reduction; Exact Match (LongBench
substring protocol) drops from $71.4\%$ to $33.3\%$, reflecting a genuine
quality loss that the F1 gain only partially offsets. On RepoBench-C across
four models ($n = 200$), Vec+F achieves mean Edit Similarity $0.683$ versus
baseline $0.722$ ($\Delta = -0.039$) at $\mathbf{54.7\%}$ input token
reduction; Mistral Small 24B shows only $\Delta = -0.026$ at $54.4\%$
reduction. The optimal fact block strategy varies by model architecture:
LLaMA-3 benefits from model-generated summaries while Qwen2.5, DeepSeek, and
Mistral benefit from structured code-fact retrieval. Larger models exhibit
smaller degradation under compression, suggesting that residual stream encoding
scales favorably with model capacity.
\end{abstract}

\section{Introduction}

Modern language model deployments increasingly follow a retrieval-augmented
generation (RAG) pattern: for each user query, a long context document $\ctx$
is retrieved and prepended to the prompt. At context lengths of 1,000--8,000
tokens, $\ctx$ dominates inference cost --- attention prefill scales at
$O(|T(\ctx)|^2)$ in the full-attention case, and $O(|T(\ctx)|)$ even with
linear-attention variants. For a single query, this cost is unavoidable. For
\emph{repeated} queries over the same context --- a product manual, a codebase
file, a research article --- the full cost is paid identically for every query,
even though the context has not changed.

This is the problem RSCE addresses: \emph{context without context}. If a model
has already ``seen'' $\ctx$ once, subsequent queries over the same document
should not require re-tokenizing and re-attending over the full context.
RSCE achieves this by encoding $\ctx$ into a fixed-size vector $C$
at a calibrated intermediate layer $f(M)$ of the model, then injecting $C$ into
the residual stream at query time. The model generates as if it has seen $\ctx$
--- the representation is present at the critical mid-network position where
semantic content is consolidated --- while the actual token prefill contains
only the fact block $F$ and query $P$.

The core observation is that by layer $f(M)$, the residual stream contains a
distributed semantic encoding of the entire input. Mean-pooling over token
positions yields a single vector $C \in \mathbb{R}^{d_M}$ that captures the
high-level semantic content of $\ctx$. Injecting $C$ at the same layer during
a separate generation pass provides the subsequent layers with a representation
informed by $\ctx$ without ever having tokenized it.

RSCE encodes $\ctx$ once. The resulting vector $C$ is stored and reused
indefinitely: all subsequent queries pay only $O(|T(F)| + |T(P)|)$ in input
tokens. With typical context lengths of 2,000--8,000 tokens and fact-block
sizes of 50--150 tokens, the per-query token savings exceed 95\%.

\paragraph{Contributions.}
\begin{itemize}
    \item A training-free, inference-time context compression method that
          eliminates repeated context token costs via residual stream vector
          injection (Section~\ref{sec:method}).
    \item A two-channel representation design separating semantic priming
          (latent vector $C$) from factual anchoring (explicit fact block $F$),
          with ablation showing $F$ is essential for factual QA and
          architecture-dependent for code completion.
    \item Evaluation across four decoder-only architectures on both
          multi-document QA and cross-file code completion, demonstrating
          97.8\% token reduction on QA and 54.7\% on code at competitive
          answer quality (Section~\ref{sec:results}).
    \item Evidence of a \emph{scaling trend}: Mistral 24B degrades significantly
          less than 7--14B models ($\Delta = -0.017$ vs.\ $\Delta \approx -0.04$
          to $-0.07$), with model architecture co-determining compression
          robustness alongside parameter count (Section~\ref{sec:results}).
\end{itemize}

\section{Related Work}
\label{sec:related}

\paragraph{Prompt-level compression.}
LLMLingua \cite{jiang2023llmlingua} and Selective Context \cite{li2023compressing}
compress the context by selecting or scoring tokens and discarding low-importance
content, transmitting a shorter but still explicit text sequence. These methods
operate in token space and must transmit the compressed text to the model at
every query. RSCE operates in representation space: the context is not compressed
into a shorter text but encoded into a latent vector, reducing per-query token
cost to $O(|F| + |P|)$ regardless of context length.

\paragraph{KV cache eviction.}
H2O \cite{zhang2023h2o}, StreamingLLM \cite{xiao2023streamingllm}, and SnapKV
\cite{li2024snapkv} evict low-importance KV cache entries during a single
generation pass to reduce peak memory. These methods compress \emph{within} one
inference call; they do not eliminate the initial full-context prefill, and the
compressed cache is not reusable across separate calls. RSCE's encoded vector
$C$ persists across arbitrary future inference calls and requires no prefill
of $\ctx$ at all at query time.

\paragraph{Learned context compression.}
ICAE \cite{ge2023icae} and AutoCompressors \cite{chevalier2023autocompressors}
learn to compress context into soft summary tokens via fine-tuning a specialized
encoder. GIST tokens \cite{mu2023learning} learn to compress instructional
prefixes. These methods achieve strong compression but require training new
model parameters or fine-tuning existing ones. RSCE requires no training:
it uses the model's own residual stream, which already encodes semantic content
as a byproduct of the standard forward pass.

\paragraph{Activation steering and representation engineering.}
Activation addition \cite{turner2023activation} and representation engineering
\cite{zou2023representation} modify activations to steer model behavior along
learned concept directions, typically derived from contrastive prompt pairs.
The target is model \emph{behavior} (e.g., make the model more helpful). RSCE
targets \emph{external document content}: the injected vector $C$ is the
mean-pooled activation over all token positions of $\ctx$, encoding what $\ctx$
says rather than how the model should behave.

\paragraph{Soft prompts and prefix tuning.}
Prefix tuning \cite{li2021prefix} and prompt tuning \cite{lester2021power} prepend
learnable soft tokens to the input sequence to condition model behavior. These
tokens occupy positions in the attention stream and interact with subsequent
tokens through attention. RSCE injects into the \emph{residual stream} at an
intermediate layer, not at token positions, and uses activation values derived
from an actual document rather than learned parameters.

\paragraph{Context distillation.}
Context distillation \cite{askell2021general} fine-tunes a model on
(context, response) pairs so that context knowledge is absorbed into model
weights and need not be supplied at inference. This is a training-time operation
and permanently modifies the model for a specific knowledge base. RSCE is
\emph{ephemeral}: $C$ is computed and injected at runtime, the model weights are
unchanged, and the same model can serve queries over any context by substituting
$C$.

In summary, RSCE is distinguished by being (1) training-free, (2) cross-pass
and indefinitely reusable, (3) residual stream injection at a calibrated
intermediate layer, and (4) coupled with a two-channel factual augmentation
design.

\section{Formal Specification}
\label{sec:formalism}

Let $M$ be a decoder-only transformer model, $T$ a tokenizer, and $P$ a prompt
string.

\paragraph{Optimal injection layer.}
\[
    f(M) \;\rightarrow\; L \in \mathbb{Z}
\]
The layer in $M$ at which to inject the context vector. Determined empirically
per architecture via calibration sweep (Section~\ref{sec:calibration}).

\paragraph{Residual context encoding.}
\[
    g(M, \ctx) \;\rightarrow\; C \in \mathbb{R}^{d_M}
\]
where $d_M$ is the hidden dimension of $M$. Specifically:
\[
    g(M, \ctx) \;=\; \mpool\!\left(\res_M\!\left(\ctx,\, f(M)\right)\right)
    \;=\; \frac{1}{|T(\ctx)|}\sum_{i=1}^{|T(\ctx)|} H_i
\]
where $H \in \mathbb{R}^{|T(\ctx)| \times d_M}$ are the residual stream hidden
states at layer $f(M)$. $C$ is a fixed-size latent representation independent
of $|\ctx|$.

\paragraph{Fact extraction.}
\[
    h(D) \;\rightarrow\; F \in \mathcal{V}^*
\]
Extracts a compact explicit token sequence $F$ from knowledge base $D$.
$F$ preserves factual precision that $C$ alone cannot provide due to lossy
mean-pooling.

\paragraph{Baseline inference.}
\[
    \gen\!\left(M,\; T(\ctx \oplus F \oplus P)\right)
    \qquad
    \text{input tokens: } |T(\ctx)| + |T(F)| + |T(P)|
\]

\paragraph{RSCE inference.}
\[
    \geni\!\left(M,\; T(F \oplus P),\; C,\; f(M)\right)
    \qquad
    \text{input tokens: } |T(F)| + |T(P)|
\]
where $\geni$ adds $C$ to the residual stream at layer $f(M)$ for every
token position: $H'_i \leftarrow H_i + \alpha \cdot C$.

\paragraph{Token savings.}
\[
    \Delta \;=\; |T(\ctx)| \qquad \text{(strictly positive for any non-empty context)}
\]

\paragraph{Quality bound.}
For any decoder transformer $M$ and context $\ctx$, there exists a layer
$f(M)$ and tolerance $\epsilon_M$ such that:
\[
    \left|\;\qual\!\left(\geni(M,\; T(F \oplus P),\; g(M,\ctx),\; f(M))\right)
    \;-\; \qual\!\left(\gen(M,\; T(\ctx \oplus F \oplus P))\right)\right|
    \;\leq\; \epsilon_M
\]
The existence of $f(M)$ for each architecture and the empirical magnitude of
$\epsilon_M$ are the central claims this work evaluates.

\section{Method}
\label{sec:method}

\subsection{Residual Context Extraction}

Given context $\ctx$ and target layer $L = f(M)$, we perform a single
forward pass of $M$ on $T(\ctx)$, intercept the residual stream at layer $L$,
and compute the token-position mean:
\[
    C = \frac{1}{|T(\ctx)|} \sum_{i=1}^{|T(\ctx)|} H_i
    \;\in\; \mathbb{R}^{d_M}
\]
No gradients are required. The pass runs on the same hardware and model used
for generation, completing in approximately one baseline inference call in latency.

\subsection{Residual Stream Injection}

At query time, the model receives token input $T(F \oplus P)$ only --- the
full context is absent from the token stream. At layer $L = f(M)$, before
the standard layer computation, $C$ is added to the residual stream at all
token positions:
\[
    H'_i \;\leftarrow\; H_i + \alpha \cdot C \qquad \forall i
\]
with scale factor $\alpha = 1.0$ (default). The modified residual stream
is processed by the remaining $n_\text{layers} - L$ layers to produce the
output distribution. The injection is applied once per autoregressive step.

\subsection{Fact Block Construction}
\label{sec:fblock}

The fact block $F$ provides an explicit textual anchor for hard-to-reconstruct
facts. We implement two variants:

\textbf{NER/code-fact extraction} (\texttt{Vec+F}): For QA tasks, we extract
named entities and numerical values from $\ctx$ using a rule-based extractor.
For code completion, we extract function signatures, class names, and key
constants from cross-file imports using BM25-ranked retrieval over the codebase.
The extracted items are formatted as a compact ``\texttt{Facts:}'' prefix block.

\textbf{Model summary} (\texttt{Vec+F\textsubscript{sum}}): A one-time
generation pass produces a concise prose summary of $\ctx$, which is used as
$F$ for all subsequent queries. This variant incurs higher setup cost (two
model passes instead of one) but may better capture relational context.

\subsection{Injection Layer Calibration}
\label{sec:calibration}

For each model $M$, we determine $f(M)$ by an accuracy sweep over a 10-example
calibration set drawn from HotpotQA and 2WikiMultiHopQA. For each candidate
layer $L$ in $[n_\text{layers}/4,\; 0.85 \cdot n_\text{layers}]$ at stride 2:
\begin{enumerate}
    \item Extract $C$ at layer $L$ from each calibration example's context.
    \item Generate answers using $F \oplus P$ (NER fact block) with injection at $L$.
    \item Compute average Token F1 against gold annotations.
\end{enumerate}
The layer with highest average F1 is selected as $f(M)$.

Table~\ref{tab:calibration} reports the calibrated $f(M)$ per architecture.
For LLaMA-3, Qwen2.5, and Mistral, the same calibrated layer was used in
both the QA and RepoBench-C evaluations. For DeepSeek-R1-Distill, the QA
calibration identified layer 12 (25\% depth) as optimal; the RepoBench-C
evaluation used layer 29 (60\% depth) based on a separate code-task
calibration, reflecting that the optimal injection layer can differ across
task types for the same model.

F1 (0.085) despite 14B parameters is consistent with its chain-of-thought
distillation training objective, which may distribute representations across
layers differently than direct-answer models.

\section{Experimental Design}
\label{sec:design}

\subsection{Motivation for Dual Evaluation}

We evaluate on two distinct domains to probe complementary properties of
residual stream encoding. \textbf{Multi-document QA} tests the semantic
retrieval aspect: the model must reason across facts distributed throughout
a long context. The residual vector must encode who-did-what-when information.
\textbf{Cross-file code completion} tests structural knowledge retention:
the model must continue a line of code consistent with functions, classes, and
constants defined in cross-file imports. The residual vector must encode API
signatures and naming conventions. Success in both domains provides stronger
evidence that RSCE is a general-purpose compression mechanism rather than
a task-specific artefact.

\subsection{Models and Hardware}

We evaluate on four publicly available instruction-tuned models at 4-bit
quantization via the MLX framework on Apple Silicon (M-series):

\begin{itemize}
    \item \textbf{LLaMA-3 8B Instruct} (32 layers, $d_M = 4096$)
    \item \textbf{Qwen2.5 7B Instruct} (28 layers, $d_M = 3584$, GQA)
    \item \textbf{Mistral Small 24B Instruct} (40 layers, $d_M = 5120$, sliding-window)
    \item \textbf{DeepSeek-R1-Distill 14B} (48 layers, $d_M = 5120$, CoT-distilled)
\end{itemize}

\subsection{Task 1: Multi-Document QA (LongBench)}

We sample from HotpotQA and 2WikiMultiHopQA via the THUDM/LongBench dataset,
filtering to context lengths 200--4,000 words. The evaluation uses $n = 42$
examples on LLaMA-3 8B Instruct. Context lengths range from approximately
500--3,500 words (650--4,500 tokens). Average baseline input length is
\textbf{4,340 tokens}; average Vec+F input length is \textbf{97 tokens}.

\paragraph{Conditions evaluated.}
\begin{itemize}
    \item \textbf{Baseline}: Full context $\ctx$ + fact block $F$ + query $P$
          in the token stream.
    \item \textbf{Vec}: RSCE vector injection; no explicit fact block ($F = \varnothing$).
    \item \textbf{Vec+F}: RSCE vector injection with NER fact block.
\end{itemize}

\paragraph{Metrics.} Token F1 (SQuAD-style token overlap against any gold
answer) and Exact Match (EM: gold answer string appears in generated output,
case-insensitive). Both follow the LongBench evaluation protocol.

\subsection{Task 2: Cross-File Code Completion (RepoBench-C)}

We use the RepoBench-C benchmark \cite{liu2023repobench}, specifically the
\texttt{cross\_file\_first} split at the \texttt{2k} token level. RepoBench-C
tests next-line code completion where the relevant context appears in a
different file from the completion target. The benchmark provides cross-file
context snippets and expects the model to generate the next line of code
consistent with the codebase structure.

We evaluate $n = 50$ examples per model across all four architectures
(200 total). Injection layers match the QA calibration values for LLaMA, Qwen,
and Mistral; for DeepSeek, layer 29 (60\% depth) was used in the code
experiments based on a separate code-task calibration.

\paragraph{Conditions evaluated.}
\begin{itemize}
    \item \textbf{Baseline}: Full in-file context + cross-file context + query.
    \item \textbf{Vec}: RSCE vector over the cross-file context; in-file context
          retained in the token stream.
    \item \textbf{Vec+F}: RSCE vector + code fact block (function signatures,
          class names extracted from cross-file imports).
    \item \textbf{Vec+F\textsubscript{sum}}: RSCE vector + model-generated
          summary of cross-file context.
\end{itemize}

In RepoBench, the \emph{cross-file} context is compressed by RSCE while the
\emph{in-file} context (the local code being edited) remains in the token
stream. This reflects the practical scenario where local context is always
available but cross-file imports are expensive to include.

\paragraph{Metric.} Edit Similarity (EditSim): \texttt{difflib.SequenceMatcher}
ratio between the model's generated next line and the gold next line, averaged
over examples. Higher is better; baseline establishes an upper bound for
full-context generation.

\subsection{Amortization Framework}

Let $T_\text{setup}$ be the token cost of the encoding pass (one forward pass
over $\ctx$, $\approx |T(\ctx)|$ tokens). Let $T_\text{query}$ and
$T_\text{baseline}$ be the per-query token costs for RSCE and baseline
respectively. The break-even number of queries is:
\[
    N^* = \frac{T_\text{setup}}{T_\text{baseline} - T_\text{query}}
\]
For QA ($T_\text{baseline} = 4{,}340$, $T_\text{query} = 97$, $T_\text{setup} = 4{,}340$):
\[
    N^* = \frac{4{,}340}{4{,}340 - 97} \approx 1.02
\]
RSCE amortizes its setup cost after just \textbf{one query}. Every subsequent
query saves 97.8\% of input tokens.

\section{Findings}
\label{sec:results}

\subsection{Token Efficiency}

By construction, RSCE eliminates $|T(\ctx)|$ tokens from every query prompt.
The reduction is model-agnostic and holds regardless of whether answers are
correct. Table~\ref{tab:efficiency} summarizes token efficiency across domains.

\begin{table}[h]
\centering
\caption{Input token reduction by domain and condition. Averages over all
examples per domain.}
\label{tab:efficiency}
\begin{tabular}{lrrrr}
\toprule
Domain & Avg baseline tokens & Avg Vec+F tokens & Reduction \\
\midrule
QA (LLaMA-3 8B)    & 4,340 & 97  & 97.8\% \\
Code (RepoBench, avg) & 984  & 445 & 54.7\% \\
\bottomrule
\end{tabular}
\end{table}

The smaller code reduction (54.7\% vs. 97.8\%) reflects the RepoBench structure:
the full in-file context remains in the token stream. Only the cross-file context
tokens are eliminated. The QA reduction is near-total because the full document
context is compressed.

\label{sec:qa_results}

\bottomrule
\end{tabular}

Three findings stand out.

\textbf{Vec alone achieves F1 = 0.000.} Without the explicit fact block, vector
injection produces no measurable F1 on this evaluation. Multi-hop QA requires
precise entity recall (specific names, dates, quantities), and mean-pooled
activations cannot reconstruct individual token identities. \emph{Caveat:}
we do not have a question-only baseline (no context, no vector); we therefore
cannot determine whether the vector injection contributes any semantic priming
above zero or whether F1 = 0 reflects pure guessing. What we can confirm is
that the NER fact block is necessary for non-zero performance, motivating the
two-channel design.

\textbf{Vec+F exceeds baseline on aggregate F1.} With the NER fact block,
Vec+F achieves F1 = 0.218 versus baseline 0.165. On HotpotQA specifically,
Vec+F achieves 0.386 versus 0.238 --- a 62\% relative improvement. We attribute
this to the SQuAD-style F1 metric, which rewards answer precision. The baseline
model, given the full context, tends to generate verbose responses that include
the answer embedded in a longer passage; the SQuAD F1 penalizes these by
dividing token matches by the full output length. Vec+F generates shorter,
more focused responses, achieving higher precision at similar recall.

\textbf{EM drops substantially.} The EM metric checks for exact answer string
containment in the output. Baseline EM (71.4\%) is high because verbose baseline
outputs usually contain the gold answer string somewhere. Vec+F EM (33.3\%)
reflects two effects: fewer exact gold strings in compact outputs, and
occasional paraphrasing. The EM gap on 2WikiMultiHopQA (64.0\% vs.\ 20.0\%)
is larger, suggesting multi-hop reasoning is harder to recover from compressed
representations than single-hop QA.

\textbf{Interpretation.} The F1/EM divergence reflects a real trade-off:
RSCE compresses context into a form that enables precise focused answers (high
F1) but loses exact answer-containment in verbose outputs (lower EM). Whether
this is a genuine quality improvement depends on the downstream application.
For information extraction or search, shorter precise answers (Vec+F) are
preferable; for verbatim answer-in-output requirements (certain QA evaluation
protocols), baseline is superior on EM.

near-lossless. The fact block contributes $+0.059$ EditSim for DeepSeek, far
exceeding its benefit for other models ($\leq +0.008$). DeepSeek's chain-of-thought
training may distribute API-signature knowledge differently across layers,
making explicit code-fact retrieval especially valuable.

\subsection{Amortization Analysis}

For QA, the break-even point $N^* \approx 1.02$ queries (see
Section~\ref{sec:design}). Every query past the first saves 97.8\% of input
tokens. For code (RepoBench level), with average baseline 984 tokens and Vec+F
445 tokens, $N^* = 984 / (984 - 445) \approx 1.8$ queries. In practical RAG
deployments, the same context is queried dozens to thousands of times, placing
all realistic use cases deep into the amortized regime.

The setup cost can be reduced further by early-exit extraction: since we only
need the residual stream at layer $f(M)$, the encoding pass can halt at layer
$f(M) + 1$ rather than completing the full forward pass, reducing setup cost
to approximately $(f(M)/n_\text{layers}) \times T_\text{baseline}$ tokens.
For Mistral at 25\% depth, this reduces setup cost to $0.25 \times N^*$,
effectively making $N^* \approx 1.0$ for all models.

\section{Discussion}
\label{sec:discussion}

\subsection{The Two-Channel Architecture}

The most important finding from the QA evaluation is that vector injection
alone is insufficient for factual question answering: Vec achieves F1 = 0.0
while Vec+F achieves 0.218. This motivates the two-channel view of context
knowledge: (1) \emph{semantic priming} --- topical, relational, and temporal
context that the model needs to reason correctly --- is well-captured by the
mean-pooled residual vector $C$; (2) \emph{factual anchoring} --- specific
names, dates, quantities, and API identifiers that must be retrieved exactly
--- requires explicit tokens in the attention stream.

Mean-pooling over token positions is a lossy operation that preserves global
semantic structure but destroys the identity of individual tokens. A context
document about Marie Curie will produce a vector $C$ that orients the model
toward ``physics, radioactivity, Nobel Prize'' at a high level, but $C$ cannot
reconstruct ``Marie Sk\l{}odowska-Curie'' as a specific string. The fact block
provides the specific string while $C$ provides the topical context for
reasoning about it.

This separation has a practical implication: the fact block $F$ need not be
exhaustive. Including the top-$k$ named entities or function signatures covers
the ``needle-in-a-haystack'' retrieval cases, while $C$ handles the broader
reasoning context. The typical fact block in our experiments is 50--150 tokens,
which is sufficient for high F1 while keeping per-query cost low.

\subsection{Code Context as Primary Deployment Target}

The QA evaluation demonstrates that the method works, but code context is the
more practically compelling target. In software development, a model working
on a codebase repeatedly queries the same set of cross-file imports, class
definitions, and utility functions. These files change infrequently relative
to the number of queries (e.g., a module's public API is stable across hundreds
of developer interactions). Under these conditions:

\begin{enumerate}
    \item A codebase can be pre-encoded once per file or per module.
    \item Each developer query pays only for the local file context and the
          question, not for the full cross-file context.
    \item When a file is modified, only its cached vector $C$ is updated ---
          a single re-encoding pass.
\end{enumerate}

At the RepoBench 2k token level, Vec+F achieves 54.7\% cross-file token
reduction with only $-0.039$ mean EditSim degradation (3 of 4 models within
$-0.061$ or better). For Mistral 24B, the degradation is $-0.026$ ---
within measurement noise for many applications. A production deployment
using Mistral-class models would lose approximately 3.4\% of code completion
quality while halving the effective context window cost on cross-file lookups.

\subsection{Architecture-Specific Behaviour}

Three architecture-specific observations are worth noting for future work.

\textbf{Sliding-window attention (Mistral)} consolidates global context early.
Mistral's optimal injection layer at 25\% depth with the highest calibration F1
suggests that sliding-window attention produces richer layer-20--layer-10
representations for external documents than full-attention models do.

\textbf{Chain-of-thought distillation (DeepSeek)} distributes context differently.
DeepSeek shows the largest Vec degradation but also the largest F-block gain.
Its residual stream at layer 29 (60\% depth) may encode structural code patterns
(API usage) less directly than semantic patterns, making explicit fact retrieval
more valuable.

\textbf{Grouped-query attention (Qwen)} peaks at later layers. Qwen2.5's
optimal injection at 61\% depth suggests that GQA requires deeper processing
to consolidate multi-document context into a query-accessible representation.

These patterns suggest that $f(M)$ calibration is not merely a hyperparameter
but reflects genuine architectural differences in how and where context is
consolidated within the residual stream.

\subsection{Limitations}

\paragraph{QA evaluation scope.}
The full QA benchmark was run on LLaMA-3 8B only ($n = 42$). Cross-model QA
results would strengthen the generalization claim. The QA F1 improvement may
not replicate on other architectures given the observed model-specificity of
F-block strategy.

\paragraph{Missing question-only baseline.}
The QA evaluation lacks a question-only condition (no context, no vector). Without
it, we cannot quantify whether the Vec-only F1 = 0 reflects a failure of vector
injection specifically, or merely a hard lower bound that any no-context condition
would achieve. A Q-only baseline would isolate the vector's contribution.

\paragraph{Lossy mean-pooling.}
Mean-pooling discards sequential structure. Tasks requiring precise ordering
of events or fine-grained entity disambiguation within long documents (e.g.,
``which event happened first?'') may not be addressable from $C$ alone.

\paragraph{Uniform residual stream injection.}
$C$ is added uniformly to all token positions at layer $f(M)$. This imposes
a single global shift rather than position-specific conditioning. Richer
injection schemes (e.g., cross-attention over a sequence of extracted context
vectors, or position-dependent scaling) may preserve more structure.

\paragraph{4-bit quantization.}
All evaluations use 4-bit MLX quantization. Quantization alters residual stream
geometry and may inflate or deflate compression quality relative to
full-precision baselines. This is a platform constraint, not a fundamental
limitation; full-precision experiments are left for future work.

\paragraph{Small-$n$ benchmarking.}
RepoBench evaluations use $n = 50$ per model. Statistical confidence intervals
are not reported. The EditSim differences (particularly for the $\Delta \approx
-0.008$ DeepSeek result) should be interpreted cautiously until confirmed on
larger samples.

\section{Conclusion}
\label{sec:conclusion}

We have presented RSCE, a training-free method for substituting repeated
long-context token costs with a one-time residual stream encoding and
per-query vector injection. The method requires no model modification,
operates across four distinct decoder-only architectures, and amortizes
fully after $N \approx 1$--$2$ queries.

On multi-document QA with LLaMA-3 8B, Vec+F achieves Token F1 of 0.218 versus
baseline 0.165 at 97.8\% input token reduction, with the NER fact block being
essential to recovery from lossy compression. On RepoBench-C code completion
across four models (7B--24B), Vec+F achieves mean EditSim 0.683 versus baseline
0.722 ($\Delta = -0.039$) at 54.7\% token reduction. Mistral Small 24B
approaches near-lossless code compression ($\Delta = -0.026$), and DeepSeek-R1
shows the largest fact-block benefit ($+0.059$ from code-fact extraction).
A scaling trend is visible: larger models degrade less under compression,
suggesting that residual stream encoding quality improves with model capacity.

The central conjecture --- that $|\epsilon_M|$ is small enough for practical
use --- holds across architectures at the RepoBench 2k code level, and
conditionally on HotpotQA QA (F1 improves; EM degrades). Future work should
extend to: longer contexts (8K--32K tokens), full-precision evaluation,
cross-model QA replication, and richer injection mechanisms beyond scalar
global shift.
